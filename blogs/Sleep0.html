# 源码中的 Thread.sleep(0)

老版本的 RocketMQ，Dubbo 的源码中，都出现了 Thread.sleep(0)，这行代码有什么用呢？

先说结论：让线程进入「安全点」，让GC线程可能被执行，避免STW过久

削峰填谷的思想，很长时间进行一次GC可能导致STW过久，因此主动在大循环内部发起GC

STW Stop-The-World：就是停顿所有线程，CMS做GC的第一、三步骤都需要STW

我们首先来了解下面两个概念：

# 安全点

特定的某些位置。当线程运行到这类位置时，堆对象状态是确定一致的。

安全点是在程序执行期间的所有GC Root已知并且所有堆对象的内容一致的点。

如果要触发一次GC，那么JVM里的所有Java线程都必须到达GC safepoint。

HotSpot中，安全点位置主要在：

  1. 方法返回之前
  2. 调用某个方法之后
  3. 抛出异常的位置
  4. 循环的末尾

目的就是避免程序长时间无法进入safepoint

如果有线程一直没有进入到安全点，就会导致GC时JVM停顿时间延长。比如写了一个超大的循环导致线程一直没有进入到安全点，GC前停顿了8秒。

# 可数/不可数循环

for ( int i = 0; i <= n ; i++ )

可数循环就是循环变量是个int或更小，不可数循环是long。

**可数循环仅在循环结束后有安全点。**因为安全点过多也会影响性能。

所以如果担心大循环延缓了GC线程，只需要改成不可数循环即可解决。

那当然另一种方法，就是调用 Thread.sleep(0)。

# 为什么Thread.sleep(0)也能解决

一个线程在运行 native 方法后，返回到 Java 线程后，必须进行一次 safepoint 的检测。 native 方法不受JVM管理，因此JVM就认为该线程进入了safepoint。

而Thread.sleep(0)正是一个native 方法。

因此调用 sleep 方法的线程会进入 Safepoint。

# 安全点+可数循环引发的问题

下面这段代码，主线程会等待t1,t2执行完才输出num

public class MainTest {
    public static AtomicInteger num = new AtomicInteger(0);
    public static void main(String[] args) throws InterruptedException {
        Runnable runnable=()->{
            for (int i = 0; i < 1000000000; i++) {
                num.getAndAdd(1);
            }
        };
        Thread t1 = new Thread(runnable);
        Thread t2 = new Thread(runnable);
        t1.start();
        t2.start();
        Thread.sleep(1000);
        System.out.println("num = " + num);
    }
}

原因：JVM会周期性地让所有线程进入安全点。

周期由 -XX:GuaranteedSafepointInterval 选项控制，该选项默认为 1000ms。

主线程sleep了1000ms，两个大循环执行也远大于1000ms，而sleep和循环结束都是安全点

因此主线程继续阻塞等待其他java线程执行循环完毕进入安全点。

# 为什么要进入安全点

VMThread ：JVM内部线程，负责协调其它线程达到安全点以及执行 VM 操作。

VM 操作是什么？

CMS 的初始标记和最终标记都是 ，又比如 thread dump，线程挂起以及偏向锁的撤销等等都是 VM 操作。

VMThread 里面有个 VMOperationQueue 队列，用于存放一个一个连在一起的 VM 操作。loop() 方法是 VMThread 的核心方法，该方法不断从 VMOperationQueue 队列中获取待执行的 VM 操作，然后调用每种 VM 操作具体的实现 evaluate() 方法执行不同的逻辑。达到安全点后还会判断是否有stub 缓存要清理

stub 缓存 : JVM 执行期间的编译解释代码缓存

关掉-XX:GuaranteedSafepointInterval这个参数也有可能会造成一些未知错误。设置为0，不会有这个问题。

让主线程直接输出num的解决方案：

  • sleep100ms
  • 修改JVM参数，周期暂停参数设置为2000ms
  • -XX:+UseCountedLoopSafepoints 选项可以关闭安全点轮询的优化。也就是在可数循环中也会有安全点。
  • JDK10以上版本实现 Loop Strip Mining 优化，解决了在 counted loop 中安全点轮询的问题，而且没有太多开销，也会按照预期输出
  • long / sleep(0)方案

# 小结

在大循环里执行Thread.sleep(0)，使得该线程进入安全点，避免了GC STW过久的问题。但更简洁的做法是把循环变量从 int 改为 long。
