# 底层实现原理：quicklist双向链表

Redis链表与传统数据结构中学到的链表不太相同。

# 普通链表存在的问题

普通的链表如下所示：

有一个潜在的问题是：浪费空间。比起数组，双向链表的pre，next指针会浪费大量空间，而Redis作为一个基于内存的数据库，这个开销有点太大了。一旦data本身也比较小，那很容易出现内存空间利用率低，大部分内存都用于存储指针的情况。

那如何解决这个问题呢？Redis的做法是：让data更紧凑，让指针更少。

# Redis：数组作为链表节点

quicklist实际上是 zipList（listpack） 和 linkedList 的混合体。

在 Redis 7.0 中，ziplist已经废弃了，交由 listpack 数据结构来实现了。但它们的本质都是连续的数组。

quicklist将 linkedList按段切分，每一段使用 zipList 来紧凑存储，多个 zipList 之间使用双向指针串接起来。

typedef struct quicklist { 
    quicklistNode *head;
    quicklistNode *tail;
    unsigned long count;        /* ziplist 的个数 */
    unsigned long len;          /* quicklist 的节点数 */
    unsigned int compress : 16; /* LZF 压缩算法深度 */
    // ...
} quicklist;
typedef struct quicklistNode {
    struct quicklistNode *prev;
    struct quicklistNode *next;
    //  quicklistNode -> ziplist 
    unsigned char *zl;           
    unsigned int sz;             /* ziplist 字节数 */
    unsigned int count : 16;     /* ziplist 个数 */
    unsigned int encoding : 2;   /* RAW==1 or LZF==2 */
    unsigned int container : 2;  /* NONE==1 or ZIPLIST==2 */
    // ...
} quicklistNode;
typedef struct quicklistLZF {
    unsigned int sz; 
    char compressed[];
} quicklistLZF;

# 小结

List是一个双向链表，不同的是，链表的每个节点是一个数组（ziplist/listpack），因此更加紧凑，能利用CPU缓存的同时，减少内存存储空间。

7.0版本的列表（List）的数据类型可以存储超过4GB的单个元素
