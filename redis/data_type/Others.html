# 其它数据结构简介

# 1、Bitmap：二进制数组

Bitmap 可以理解为一个特殊的数组，每个元素的值只能为 0 或者 1。

Bitmap 是基于SDS实现的，而SDS最大长度是 512M，Bitmap自然也是。

Bitmap 的 get/set 方法O(1)，BITCOUNT用 查表法 和 SWAR算法 实现，复杂度也在常数级别。

# 基本读写：SET/GETBIT

# 2、HyperLogLog：去重计数

UV，Unique Visitor，独立访客，需要根据用户ID去重，这就是 HyperLogLog 最典型的应用场景。

HyperLogLog 提供不精确的去重计数方案，可以用来做基数统计，从而实现 UV 统计功能。

HyperLogLog 内存占用极低，性能高，但缺点是有误差，标准误算率是 0.81%，在业务不需要完全精确的情况可以使用。

# HyperLogLog 常用命令

HyperLogLog 相关的命令都以 pf 开头

# 添加元素：pfadd

pfadd [HyperKey] "java" "sql"
1

# 统计不重复的元素：pfcount

pfcount [HyperKey]
1

# 合并：pfmerge

pfmerge [合并的目标HyperKey] [sourcekey1] [sourcekey2] ...

# 3、GEO：地理位置信息

GEO 可以用来表达地理位置信息。

GEO基于 ZSet 实现，使用 GeoHash 编码方法，将经纬度转化为ZSet 的权值。

# GEO常用命令

# 添加地理位置：geoadd

geoadd [GEOkey] [经度1] [纬度1] [经纬度1的value] [经度2] [纬度2] [经纬度2的value] ...
1

# 查询位置信息：geopos

geopos [GEOkey] [value1] [value2] ...
1

# 删除地理位置：zrem

zrem [GEOkey] [value1] [value2] ...
1

# 直线距离统计：geodist

# 指定单位为km 默认为m

# 表示某经纬度附近[num][km]的[GEOkey]下的所有GEO的value

# Stream vs MQ

成熟的消息队列产品非常多，比如RocketMQ，为何要用Stream？

比如 Kafka 依赖于 Zookeeper，相比 Redis 来说，Kafka 和 RabbitMQ 更重量级，而 Redis 更轻量化。

如果分布式系统中的组件消息通信量不大，就可以考虑用 Redis Stream。

# Stream常用命令

  • xadd 添加消息；
  • xlen 查询消息长度；
  • xdel 根据消息 ID 删除消息；
  • del 删除整个 Stream；
  • xrange 读取区间消息
  • xread 读取某个消息之后的消息。
