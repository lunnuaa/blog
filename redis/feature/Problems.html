# Redis 常见问题

# 1、Redis 缓存穿透：访问不存在的数据

概念：redis和数据库都没有你要的数据，但你一直访问，最终造成数据库服务器瘫痪

重点：缓存不存在

# 布隆过滤器

布隆过滤器，Bloom Filter，用于检索一个元素是否在一个集合中，可以理解为一个0/1位数组，初始时全为0

存储数据的时时候，使用K个不同的哈希函数将这个变量映射为bit列表的的K个点，把它们置为1

判断缓存key是否存在，同样K个哈希函数，映射到bit列表上的K个点，判断是不是1：

1）如果全不是1，那么key不存在；

2）如果都是1，也只是表示key可能存在

布隆过滤器还有其它的应用，比如实现垃圾邮件过滤等

# 缓存空值

数据库给redis一个null缓存 TTL可以设置的较短

# 2、缓存雪崩：大量缓存key同时消失

缓存雪崩：大量redis的key同时过期，导致数据库压力太大

# 随机化过期

TTL赋值加一个小的随机数 比如 30MIN + 0~5 rand

// 缓存原本的失效时间
int exTime = 10 * 60;
// 随机数生成类
Random random = new Random();
// 缓存设置
jedis.setex(cacheKey, exTime+random.nextInt(1000) , value);

# 多级缓存

除了 Redis 本身的缓存，再设置一层缓存，当 Redis 失效之后，先去查询二级缓存。

# 3、缓存击穿：热点key消失

缓存击穿：热点Key消失了，导致数据库压力过大

# 互斥锁

查询缓存未命中，会加锁，等到写入缓存后才释放锁

缺点：其他线程等待时间较长，死锁风险

# 逻辑过期

不设置TTL，而是加expire字段 - 逻辑时间

发现时间过期了 相当于 缓存未命中 进行以下操作：

  1. 尝试获取互斥锁
    • 若获取成功，开启新的线程，查数据库更新缓存，释放锁
    • 若获取失败，直接进入步骤3
  3. 直接从缓存中返回过期数据（无论第二步结果如何，都不会等待锁的释放，而是直接返回）

优点：线程无需等待用户体验好

缺点：会有数据不一致性 消耗额外内存

# 4、内存碎片

Redis 使用 jemalloc 作为内存分配库，而非malloc。在分配内存时，会根据我们申请的字节数 N，找一个比 N 大，但是最接近 N 的 2 的幂次数作为分配的空间，这样可以减少频繁分配的次数。但缺点是，在 Redis 的使用过程中，会产生许多内存碎片。

# Redis自动清理内存碎片

Redis提供了一种内存碎片自动清理的方法，所谓清理就是使得所有实际使用的内存尽可能连续，紧凑。

config set activedefrag yes
1

# 自动清理的时机

有两个参数

# 内存碎片的字节数达到 100MB 时，开始清理
active-defrag-ignore-bytes 100mb
# 内存碎片空间占操作系统分配给 Redis 的总空间比例达到 10% 时，开始清理
active-defrag-threshold-lower 10

# 阻塞主线程

清理内存碎片的动作，相当于在对Redis进行写操作，此时Redis主线程是会被阻塞的。因此Redis

# 自动清理占用 CPU 时间的比例最少 25% 这是为了保证真的能够清理 不至于时间太短什么也没做就切换了

# 占用 CPU 时间的比例最大值 可以避免Redis长时间阻塞
active-defrag-cycle-max 75

# Redis内存碎片问题小结

如果物理内存资源比较紧张，为了避免内存碎片过多，可以考虑开启自动清理内存碎片功能。

当然另一种简单粗暴的方法是在访问量低的时机重启Redis

需要关注四个参数，用于控制触发时机和占用CPU资源。

# 5、bigkeys：本质是value过大

K-V。V是重点。比如点赞，帖子的点赞用户Id的list/set，会很大。百万人点赞。

# 大Key问题带来的影响

对大key的查询等操作，会严重影响性能，且因为单线程，会阻塞后续操作。

# 大Key对AOF的影响

虽然大Key问题的本质是value过大，但也会影响到AOF。

  1. AOF重写时，是类似RDB的将所有键值对记录为set k v，涉及到value
  2. AOF重写时，对大Key的修改采取写时复制策略，会复制整个大key，向OS申请大片内存空间会面临阻塞，OS需要加锁等操作，性能低。核心问题是这一点。

因此，value过大也会影响到AOF。

# 阻塞主线程的风险

bigkeys 命令也需要扫描整个 Redis 数据库，因此会造成主线程阻塞，线上环境可以用在从库。

另外，可以通过 -i 参数指定扫描间隔

# bigkeys不全

bigkeys 根据数据类型，返回最大的bigkey，并且统计的是集合元素个数的多少，而不是实际占用的内存量。

所以 bigkeys 的结论”仅供参考“，并不可靠。

# 2、手动scan

scan是一种安全的遍历数据库的方式。

scan [cursor] MATCH [pattern] [COUNT count]
# count:限定服务器单次遍历的字典槽位数量（约等于）
1
2

再结合 MEMORY USAGE 命令，查询一个键值对占用的内存空间

MEMORY USAGE [key]
1

从而找到实际占用内存较大的key

# 解决方案

# 对于能删的key

Redis 4.0 提供了 unlink 命令，可以异步非阻塞删除 key

# 对于不能删的key

可以对字符串压缩，拆分，对list/set分片，把一个大 key 转化成多个小 key

《有赞透明多级缓存解决方案（TMC）》中说到，有赞改写了jedis原生的jar包，加入了Hermes-SDK包，做热点发现和本地缓存，对于Jedis-Client的每次key值访问请求，Hermes-SDK 都会通过其通信模块将key访问事件异步上报给Hermes服务端集群，以便其根据上报数据进行“热点探测”。

# 请求分散
